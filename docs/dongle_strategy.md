# MonoMeasure 加密狗（硬件许可）策略建议

> 目标：在不影响正常用户体验的前提下，提高逆向和仿冒成本。

## 1. 总体架构（分层校验）

- **本地硬件因子**：USB 加密狗序列号、设备证书、公钥指纹、挑战应答能力。
- **软件环境因子**：机器指纹（CPU/主板/磁盘摘要）、系统版本、时钟可信度。
- **服务端因子（可选）**：在线激活/吊销、租期控制、并发席位控制。
- **功能因子**：按模块授权（基础测量、批量分析、导出 MP4 等）。

建议采用“**离线可运行 + 周期性在线回执**”混合模式。

## 2. 加密狗证书链与密钥体系

- 根密钥仅离线保存，用于签发厂商 CA。
- 每批次加密狗内置设备私钥（不可导出）和设备证书。
- 软件内只保留公钥链，进行证书链验证与 CRL/黑名单检查。
- 所有授权文件（license blob）使用厂商私钥签名，客户端只验签不解密私钥。

## 3. 启动与运行时校验

1. 启动时：
   - 检查加密狗在位；
   - 执行随机挑战应答（nonce + timestamp + session id）；
   - 校验证书链和授权签名；
   - 生成本次会话 token（短时有效）。
2. 运行中：
   - 每隔 30~120 秒随机发起一次轻量挑战；
   - 检测热插拔、时钟回拨、调试器附加、关键模块完整性；
   - 失败时进入降级策略（只读/禁止导出/延时退出）。

## 4. 反破解设计（建议）

- **多点校验**：将授权判断散布到关键路径（播放、Analyze、Track、Export），而非单点 if。
- **控制流混淆**：关键授权逻辑拆分为多个小函数，跨模块调用。
- **完整性保护**：关键 DLL/EXE 做哈希签名，运行时抽查。
- **反调试/反注入**：检测常见调试器、API Hook、内存补丁行为。
- **蜜罐字段**：埋设伪授权字段，若被篡改直接触发告警。

## 5. 授权模型

- 授权字段建议包括：
  - 产品/版本范围
  - 功能位图（Feature Flags）
  - 到期时间（perpetual / subscription）
  - 最大并发实例数
  - 客户标识与项目标识
- 建议支持“试用授权”、“临时离线授权”、“正式永久授权”。

## 6. 吊销与应急

- 支持本地黑名单文件（签名）+ 在线吊销列表。
- 丢失/泄漏设备可快速吊销并重新签发。
- 提供应急码（短时）用于现场抢修，但必须可审计。

## 7. 与 MonoMeasure 的落地建议

- 抽象 `LicenseManager` 接口层：
  - `bool init()`
  - `bool hasFeature(Feature f)`
  - `LicenseState state()`
  - `QString lastError()`
- 在以下动作前统一做功能位判断：
  - Analyze Particles
  - Track
  - Export CSV / BMP / MP4
- 在状态栏显示授权状态（有效/即将过期/已过期/离线宽限期）。

## 8. 迭代路线

- **V1**：本地加密狗 + 证书链 + 功能位控制 + 启动校验。
- **V2**：运行时随机挑战 + 完整性抽查 + 黑名单机制。
- **V3**：在线租期/席位管理 + 风险评分 + 自动告警。

---

该策略重点是“分层、分散、可审计”。没有任何单一手段能完全防破解，但组合策略可显著提升攻击成本。
